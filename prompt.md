# Fly Ruler Proto v1 Design Baseline

## Project Intent

Fly Ruler Proto provides the shared data kernel and client protocol for flight
simulation visualization. The 3D engine, such as Godot or Unity, is a pure
frontend: it renders world scenery, aircraft models, HUD, and UI. It does not
own flight dynamics or simulation logic.

The Rust kernel owns only:

- UDP/protobuf message ingestion.
- Heartbeat-backed client session visibility.
- In-memory time-series state and event storage.
- Explicit save/load snapshots for later inspection or playback by a frontend.
- Narrow query APIs for latest state, state ranges, event ranges, and aircraft IDs.

The kernel does not own replay, interpolation, render timing, aircraft model
loading, UI component binding, simulation stepping, or custom field semantics.

## v1 Architecture Decisions

- Runtime shape: embedded Rust library loaded by Godot/Python bindings.
- Wire format: protobuf generated by `prost`.
- Transport: Tokio UDP datagrams.
- Session model: application-layer `client_uuid` handshake and heartbeat.
- Persistence: explicit snapshot using `meta.json` plus parquet files.
- Schema policy: `toml_config`, `custom_fields`, and custom events are stored
  and returned as data; the kernel does not parse or validate their domain meaning.

TCP/bincode was an earlier experiment direction. It is not part of v1 and should
not be treated as the current implementation target.

## Runtime Boundaries

`KernelRuntime` is the orchestration entry point. It starts and stops the UDP
server, exposes the shared `TimeSeriesStore`, and delegates save/load/clear to
the store.

`transport` is responsible for UDP I/O, protobuf encode/decode, session
bookkeeping, and basic ACK behavior:

- `handshake` is ACKed when the protocol version matches.
- protocol version mismatch returns an error response.
- `heartbeat` is ACKed and refreshes session visibility.
- high-frequency `state_update` is best effort and is not ACKed.
- `spawn`, `despawn`, and `custom_event` are also ingested best effort in v1.

`store` is deterministic and explicit. It appends states/events, preserves
timestamp ordering, serves query APIs, and saves/loads snapshots only when asked.
Loading a snapshot replaces the current in-memory contents.

## Public Query Contract

The v1 query surface is intentionally small:

- `get_aircraft_ids()`
- `get_latest(aircraft_id)`
- `get_states_range(aircraft_id, start, end)`
- `get_events_range(aircraft_id, start, end)`

There is no kernel-level `at_time`, interpolation, playback cursor, pause,
resume, replay speed, or autosave behavior. A visualization frontend can build
those features by loading data into the store and querying ranges/latest state.

## Binding Roles

Python provides an aircraft-bound client abstraction. One client instance owns
one aircraft lifecycle: connect, handshake, spawn, update state, create custom
events, despawn, and close. Close performs best-effort despawn.

Godot provides an embedded server wrapper around `KernelRuntime`. It can start
and stop the UDP server, inspect sessions, query aircraft/time-series data, and
perform explicit save/load.

Unity is not implemented in v1. Future Unity integration should reuse the same
core API through a C ABI, C# binding, or a separate process wrapper if needed.

## Validation Baseline

Expected checks:

```bash
# Rust workspace
cargo test -p fly_ruler_proto_core
cargo test -p fly_ruler_proto_python
cargo build -p fly_ruler_proto_godot

# Full local validation (from repository root)
just check
just test
```

Core tests should cover handshake ACK, heartbeat ACK, protocol mismatch errors,
state/lifecycle/custom event ingestion, session timeout cleanup, timestamp
ordering, range/latest queries, and explicit snapshot save/load.
